# 开发计划与多代理分工

本文用于多代理并行开发时的边界约定、推荐生成顺序和交接检查。

## 分工边界

| 代理 | 负责范围 | 禁止事项 |
| --- | --- | --- |
| 集成文档子代理 | 根目录文档、`docker-compose.yml`、`docs/**`、启动说明、验收清单、接口概览 | 不修改 `backend/**`、`frontend/**` |
| 后端代理 | `backend/**`、数据库 SQL、Spring Boot 代码、认证、业务模块、缓存、消息、WebSocket、AI mock | 不改前端实现；如需调整接口，先同步 `docs/api-overview.md` |
| 前端代理 | `frontend/**`、Vue 3 页面、路由、状态管理、API 封装、WebSocket 前端接入 | 不改后端实现；如需调整接口，先同步 `docs/api-overview.md` |
| 验收代理 | 联调、测试、缺陷记录、验收报告 | 不回滚其他代理修改 |

协作原则：
- 以 `docs/api-overview.md` 作为前后端接口契约。
- 以 `docs/requirements-summary.md` 作为验收清单。
- 后端接口路径、请求头、返回结构变更时，同步更新文档后再通知前端。
- 前端若发现页面交互需要新增接口，先在文档中补充接口意图，再由后端实现。
- 任一代理只修改自己的负责目录，避免覆盖并行代理正在写的文件。

## 推荐生成顺序

1. 后端项目骨架。
2. 数据库 SQL。
3. 公共返回类、异常类、工具类。
4. Spring Security + JWT。
5. MyBatis-Plus 配置。
6. 用户模块。
7. 部门模块。
8. 字典模块。
9. 文件模块。
10. 通知模块。
11. RabbitMQ + WebSocket。
12. 审批模块。
13. 日程模块。
14. 新闻模块。
15. AI 模块。
16. 前端项目骨架。
17. 前端 Layout 和路由。
18. 登录页。
19. 各业务页面。
20. 前后端联调说明。
21. 测试数据和启动说明。

如果一次无法完成全部代码，优先生成完整项目结构、数据库 SQL、认证能力和基础 CRUD，再逐模块推进。

## 阶段计划

### 阶段 1：基础设施和工程骨架

目标：
- [ ] 根目录具备 `docker-compose.yml`，能启动 MySQL、Redis、RabbitMQ。
- [ ] 后端 Spring Boot 3 工程可启动。
- [ ] 前端 Vue 3 + Vite 工程可启动。
- [ ] README 中的本地启动路径可执行。

交付：
- [ ] 后端健康检查或 Swagger 可访问。
- [ ] 前端开发服务可访问。
- [ ] 基础依赖容器健康。

### 阶段 2：数据库与后端公共能力

目标：
- [ ] 建表 SQL 覆盖核心 15 张表。
- [ ] 统一 `Result<T>`。
- [ ] 全局异常处理。
- [ ] 参数校验。
- [ ] MyBatis-Plus 分页、逻辑删除、自动填充。
- [ ] Spring Security + JWT 登录认证。

交付：
- [ ] 登录接口返回 token。
- [ ] 未登录访问业务接口返回 `401` JSON。
- [ ] 无权限访问返回 `403` JSON。

### 阶段 3：基础业务模块

优先顺序：
1. 用户模块。
2. 部门模块。
3. 字典模块。
4. 文件模块。

原因：
- 用户和部门是权限、审批人匹配、日程参与人的基础。
- 字典支撑审批类型、状态、新闻分类、通知类型等。
- 文件支撑头像、审批附件、日程附件、新闻封面和附件。

交付：
- [ ] 基础 CRUD 和分页可用。
- [ ] 员工、管理员、审批人身份可配置。
- [ ] 部门树可缓存。
- [ ] 字典数据可缓存和刷新。
- [ ] 文件上传落盘并保存元数据。

### 阶段 4：通知链路

目标：
- [ ] RabbitMQ TopicExchange 配置完成。
- [ ] 统一 `NotificationMessage`。
- [ ] 生产者确认和 returns callback 生效。
- [ ] 消费者手动 ACK。
- [ ] 站内信落库。
- [ ] WebSocket 在线推送。
- [ ] 邮件提醒异步发送。

交付：
- [ ] 管理员发送系统通知后，员工站内信可见。
- [ ] 在线用户可实时收到 WebSocket 通知。
- [ ] 离线用户登录后可看到未读通知。

### 阶段 5：核心业务模块

审批：
- [ ] 草稿、提交、撤回、通过、驳回流程完整。
- [ ] 审批人自动匹配。
- [ ] 审批记录完整。
- [ ] 审批事件触发通知。

日程：
- [ ] 个人日程和会议日程完整。
- [ ] 参与人接受和拒绝。
- [ ] 定时提醒进入 RabbitMQ。
- [ ] 站内信、WebSocket、邮件提醒联动。

新闻：
- [ ] 草稿、发布、下架、置顶完整。
- [ ] 评论、点赞、收藏完整。
- [ ] 新闻缓存和缓存清理完整。
- [ ] 新闻发布触发通知。

### 阶段 6：AI 模块

目标：
- [ ] `MockAiServiceImpl` 默认可用。
- [ ] `TongyiAiServiceImpl` 预留配置和接口。
- [ ] AI 摘要、风险分析、新闻生成、新闻润色、日程解析、智能问答接口可用。
- [ ] AI 调用日志落库。
- [ ] 前端按角色展示 AI 功能。

交付：
- [ ] 管理员可使用全部 AI 功能。
- [ ] 员工只可使用智能问答和日程 AI 助手。
- [ ] 所有 AI 请求都有 `sys_ai_log`。

### 阶段 7：前端页面与联调

目标：
- [ ] Axios 统一封装，自动携带 token。
- [ ] `401` 自动跳转登录页。
- [ ] `403` 提示无权限。
- [ ] Pinia 管理用户、权限、通知、字典。
- [ ] WebSocket 登录后连接，断线重连。
- [ ] 页面使用统一后台管理布局。

页面顺序：
1. 登录页。
2. Layout、Sidebar、Header、Breadcrumb。
3. 首页仪表盘。
4. 用户管理。
5. 部门管理。
6. 字典管理。
7. 文件上传组件。
8. 审批列表、发起审批、待我审批、审批详情。
9. 日程列表、日历视图。
10. 新闻列表、新闻详情、新闻管理。
11. 站内信中心。
12. AI 助手页面。
13. 个人中心。

交付：
- [ ] 菜单按权限展示。
- [ ] 表格分页、筛选、刷新可用。
- [ ] 表单校验完整。
- [ ] 危险操作有二次确认。
- [ ] 提交 loading、成功失败提示完整。
- [ ] WebSocket 通知能更新铃铛和未读数。

## 联调检查顺序

1. `docker compose up -d` 启动依赖。
2. 后端启动并连接 MySQL、Redis、RabbitMQ。
3. 初始化 SQL 执行成功。
4. 登录 `admin` 成功并获得 token。
5. 前端登录页调用后端登录接口成功。
6. 前端请求头携带 `Authorization: Bearer <token>`。
7. 管理员菜单完整，员工菜单按权限收敛。
8. 用户、部门、字典 CRUD 联调。
9. 审批提交和审批通知联调。
10. 日程提醒定时任务联调。
11. 新闻发布和新闻缓存联调。
12. 文件上传、下载和业务附件联调。
13. AI mock 和 AI 日志联调。
14. 全量跑一遍 `docs/requirements-summary.md` 验收清单。

## 需求风险与待确认项

- [ ] 需求写“五个 AI 功能”，但接口层列出审批摘要、审批风险、新闻生成、新闻润色、日程解析、智能问答共 6 个操作；建议将新闻生成和润色视为一个新闻 AI 功能组，或明确为 6 个 AI 子功能。
- [ ] 审批 AI 摘要的权限描述存在口径风险：业务场景提到管理员或审批人打开审批详情可点击 AI 摘要，但权限段又说普通员工不可用。若审批人为员工，需确认是否允许其使用审批 AI 摘要。
- [ ] 默认登录账号和密码未在需求中明确，需要后端初始化 SQL 和 README 保持一致。
- [ ] 新闻评论直接展示，无审核机制，需确认是否需要敏感词过滤或管理员事后处理。
- [ ] 文件上传使用本地磁盘，生产环境需要确认持久化目录、访问 URL、安全校验和备份策略。
- [ ] RabbitMQ 消费失败可重新入队，但需设置重试上限或死信队列，避免无限重试。
- [ ] JWT 登出失效仅预留 Redis 黑名单，需确认是否必须实现。
